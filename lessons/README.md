## 怎麼用
- **犯錯、或發現過去的錯時**：在下面「教訓記錄」**最上面加一筆**（最新在上）。
- 格式（簡短即可）：
  - **日期 · 標題**
  - **現象**：發生了什麼、怎麼被發現
  - **根因**：為什麼會錯
  - **以後怎麼防**：守則／要掃哪裡／該加的自動檢查
- 若是**通則**，順手蒸餾進 `CLAUDE.md` 對應章節，並在該筆註明「已入 §X」。
- **動手前**（特別是查證內容／改資料／部署）先掃過本檔。

## 教訓記錄（最新在上）

### 2026-06 · 查迷宮別只查「存不存在／放對區」，要連「裡面的 boss 身分」一起查
- **現象**：使用者問「亞壇高原有一個黑刃 boss，你怎麼沒提？」一查，那是「尊貴者英雄墓地」入口的**黑刀刺客 field boss**（我們有墓地步驟卻沒提入口這隻）。順著查下去滾出整條 **boss 錯標鏈**：ch11-s40／ch18-s34 把「薩米爾的古英雄」誤標成「腐爛樹靈」；ch1-s3 近林洞窟把 boss 標成「亞人首領」（其實是海岸洞窟的）；ch11-s33↔s37 兩格 boss 整個對調；ch7 王室墓地封印監牢把掉落寫成「白王劍」（白王=另一隻 boss，該 evergaol 其實是石膚黑王/隕石）。
- **根因**：① 我先派的三個 audit subagent 只驗了「迷宮有沒有收錄、放對地區沒、有沒有重複」**三個維度，唯獨漏了「步驟裡標的 boss 對不對」這個維度**——覆蓋面自以為完整其實有結構性缺口。② 相似名陷阱：腐爛樹靈 vs 薩米爾古英雄、白王 vs 黑王、亞人首領（兩個洞窟都有）——光憑名字很容易標錯也很容易改錯。
- **以後怎麼防**：① 查/改迷宮內容時，**boss 身分是必驗維度**，跟「存在／區域／重複」並列，別漏。② 驗 boss 最強的證據是**掉落物**：步驟 `items` 裡的招牌掉落（薩米爾彎刀⇒薩米爾古英雄、山妖大錘⇒挖石山妖、歐赫寶劍⇒發狂鬥士、隕石⇒石膚黑王）拿來和 fextralife 對撞，假陽性／假陰性一抓一個準。③ 派 audit subagent 前先**列清楚要驗哪幾個維度**，否則它只會驗你明講的那幾項、漏掉沒講的。④ 相似名一律對 `docs/translations/*.csv` 官方解包定錨。

### 2026-06 · 多來源查證：別只摘亮點就宣稱「已 review」；下結論前先 audit 自己
- **現象**：使用者給 15 個支線連結，我先做「挑高信心的重點補」卻對外講成「既有資料大體吻合」——其實多數支線根本沒逐條比。後來逐條 audit，又揪出我自己 review 的**假陽性**：差點把**正確**的「聖樹騎士羅蕾塔」改成卡利亞城寨的「禁衛騎士羅蕾塔」（官方是**兩個不同 Boss**）；把可能正確的「瑟利亞封印鑰匙」當成錯。
- **根因**：① 用「摘要」代替「逐條 diff」還誇大覆蓋率；② 單一來源（巴哈/wiki 攻略文）的專名/順序當定論；③ 沒先查我們自己的 `detail`（假陽性常是「內容其實已存在」，如柏克、亞歷山大 ch8-s7、ch14-s23 三選一）。
- **以後怎麼防**：① **逐條 diff、不挑亮點**，覆蓋率要據實說（沒做就說沒做）；② 改專名前一律對 `docs/translations/*.csv`（官方解包）核對——同名不同物要當心（兩個羅蕾塔）；③ 斷言「我們缺 X」前先 grep 我們的 `walkthrough.json` 全文（含 detail）。④ review 完先**audit 自己的 review** 再動手改。


### 2026-06 · 部署別狂輪詢；工具呼叫要真的送出
- **現象**：每改一小點就打一次 tag 部署（版本從 1.7.0 爆衝到 1.8.x）；而且反覆在「查部署狀態」那步把工具呼叫寫成**純文字 `<invoke>` 標記**（沒真正送出，吐一段怪 XML 給使用者）。
- **根因**：高度重複的流程（觸發→等→查→jq）誘發「複製上一次」式的退化輸出；過度頻繁部署又放大了次數。
- **以後怎麼防**：**批次部署**——累積一組相關改動 → 本地四件套（§3）→ 一次部署；**部署後最多查一次 run**，本地 build 綠了就先回報「已觸發＋本地通過」；每個工具呼叫都當「第一次」慎重送出，不照抄節奏。

### 2026-06 · 查證內容要「跨所有資料層」核對
- **現象**：使用者問「日蔭城有秘密符節？」我只查 `walkthrough.json`（流程文字）就回「沒錯」，但錯誤其實在 `collection.json`／`links.json`——收集項「秘密符節（日蔭城）」被誤連到日蔭城步驟。
- **根因**：只驗了一層。神聖樹秘密符節右半其實在**白金村（阿爾比斯）**，被貼成「日蔭城」並放錯區；日蔭城（The Shaded Castle）根本沒有符節，那只是和「索爾城（Castle Sol）」名字相近的混淆。
- **以後怎麼防**：查證／修改一個東西時，**三層一起看**：`walkthrough.json`（步驟）＋ `collection.json`（收集項）＋ `links.json`（連動）。名稱相近的專名一律用 `docs/translations/places.csv` 對官方英文確認（日蔭城=The Shaded Castle、索爾城=Castle Sol）。

### 2026-06 · 跨區／多階段內容要放真實時點並掛進 quest
- **現象**：輝石龍亞杜拉的「首次遭遇」與「擊殺」被按地理硬塞進同一章（ch7），導致流程上「擊殺在前、遭遇在後」的怪序；且沒掛進菈妮支線。死龍弗爾桑克斯（菲雅支線高潮）也漏掛 quest。
- **根因**：把「新增內容」按地區 append，而非放在攻略路線真正發生的時點；忘了用 `quests[].stepIds` 聚合跨區鏈。
- **以後怎麼防**：新增步驟放「路線真實時點」；跨區鏈每步掛 `quests:[id]` 並加進該 quest 的 `stepIds`（依流程順序、同步 `count`）。已入 §4。

### 2026-06 · 分支基底必須接 origin/main，別信既有分支
- **現象**：工作分支從一條落後 `main` 約 40 個 commit 的舊分支切出，整批 UI 改動蓋在過時程式碼上（缺 logo、版本顯示、sticky 標題等）。
- **根因**：repo 當時的**預設分支不是 main**（是 `claude/elden-ring-strategy-tool-wqmtv8`），新分支從預設分支切就接到舊基底。
- **以後怎麼防**：動手前 `git fetch origin main && git rev-list --count HEAD..origin/main`（回 0 才安全），落後就先 rebase。別信分支名或既有狀態，基底自己驗。已入 §2 鐵則。
